# Deployment Guide

This project supports:

- Local Django and PostgreSQL development with Docker Compose
- Local React development with Vite outside Docker
- Django API deployment on Render using Docker
- React/Vite deployment on Vercel

## Local Docker Setup

From the repository root:

```bash
cp .env.example .env
docker compose up --build
```

The Docker services will be available at:

- Backend: `http://localhost:8000`
- Django Admin: `http://localhost:8000/admin/`

Run the frontend separately:

```bash
cd frontend
npm install
npm run dev
```

The Vite frontend will be available at `http://localhost:5173` and proxies `/api` requests to the Dockerized backend.

Create the first administrator:

```bash
docker compose exec backend python manage.py createsuperuser
```

Stop the stack with:

```bash
docker compose down
```

To also delete the local PostgreSQL data volume:

```bash
docker compose down -v
```

## Deploy the Backend and PostgreSQL on Render

The repository root contains `render.yaml`, which defines:

- A Docker-based Django web service
- A managed Render PostgreSQL database
- Automatic `DATABASE_URL` injection
- Generated Django secret key

### Blueprint deployment

1. Push the repository to GitHub.
2. Open Render and select **New → Blueprint**.
3. Connect the repository containing `render.yaml`.
4. Apply the blueprint.
5. Set these environment variables on the web service:

```text
CORS_ALLOWED_ORIGINS=https://your-project.vercel.app
CSRF_TRUSTED_ORIGINS=https://your-project.vercel.app
```

Do not include a trailing slash.

The container automatically runs migrations before starting Gunicorn.

Create the production administrator from the Render Shell:

```bash
python manage.py createsuperuser
```

The API will be available at a URL similar to:

```text
https://exalore-erp-api.onrender.com
```

## Deploy the Frontend on Vercel

1. Import the same GitHub repository into Vercel.
2. Set the project **Root Directory** to `frontend`.
3. Vercel should detect Vite automatically.
4. Add this environment variable for Production and Preview:

```text
VITE_API_BASE_URL=https://exalore-erp-api.onrender.com
```

5. Deploy the project.

`frontend/vercel.json` configures SPA rewrites so routes such as `/login`, `/items/new`, and `/sales-orders` work when opened directly.

After receiving the final Vercel URL, update the Render CORS and CSRF variables and redeploy the backend.

## Production Environment Variables

### Render backend

| Variable | Purpose |
| --- | --- |
| `DATABASE_URL` | Injected from Render PostgreSQL |
| `DJANGO_SECRET_KEY` | Generated by Render |
| `DJANGO_DEBUG` | Must be `false` |
| `DJANGO_SERVE_MEDIA` | Set `true` for this machine-task deployment so uploaded print template images can be served |
| `CORS_ALLOWED_ORIGINS` | Exact Vercel frontend URL |
| `CSRF_TRUSTED_ORIGINS` | Exact Vercel frontend URL |
| `WEB_CONCURRENCY` | Gunicorn worker count |

### Vercel frontend

| Variable | Purpose |
| --- | --- |
| `VITE_API_BASE_URL` | Public Render API origin, without `/api` |

## Uploaded Images

Render containers have an ephemeral filesystem. Django static assets are handled by WhiteNoise, but user-uploaded item images and print template header/footer images require a Render persistent disk or object storage such as Cloudinary/Amazon S3 before real production use. For this machine-task deployment, `DJANGO_SERVE_MEDIA=true` allows the backend to serve uploaded media files directly.

## Deployment Checks

Before deploying:

```bash
docker compose config
docker compose build backend
```

After deployment, verify:

1. `/admin/` loads on Render.
2. The Vercel login page loads directly and after refresh.
3. Admin login returns JWT tokens.
4. Item, quotation, and order APIs return data.
5. Anonymous API requests return `401` or `403`.
